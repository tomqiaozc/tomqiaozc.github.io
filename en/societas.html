01 · What Societas is

Societas is not a text generator; it is a general-purpose Agent for knowledge work.

Societas is built for knowledge workers. It began with high-quality Office artifacts, then expanded to executing and delivering complete tasks. It brings expertise, workflow steps, tool use, and task state into one Agent system that delivers results people can continue editing and using.

Outcome-oriented

Deliver complete, editable results

The system produces working files such as documents, presentations, data analyses, and web pages instead of returning only fragments in a chat window.

Safe execution

Work in isolated environments and the browser

Code and files are processed in isolation. When a task requires page interaction, the system reads the current page state and performs multi-step actions.

Reusable capability

Preserve expertise and successful steps

Domain knowledge and quality standards are saved as Skills, while stable steps from successful tasks are saved as Workflows for similar work later.

02 · What problem it solves

Traditional Copilot can generate content, but it is not designed to complete an entire real-world task on its own.

Societas is not about making AI write more. It closes three gaps between answering a question and completing a task.

Quality gap

Generated content is not yet a professional deliverable

A draft must still meet factual, structural, layout, and brand requirements before it becomes a result people can use and continue editing.

Reuse gap

One answer does not create reusable capability

Team expertise, quality standards, and successful steps must be preserved for later tasks instead of restarting from a prompt every time.

Execution gap

A recommendation does not complete the task

Real work also requires processing files, invoking tools, gathering information across pages, checking the result, and delivering the final artifact.

In one sentence: Move AI from one-off answers to reusable capability, and from content generation to task completion.

04 · TDD methodology

Turn expert judgment into reusable quality standards.

Here, TDD means Taste-Driven Development, not test-driven development. It turns brand, layout, narrative, and domain requirements into explicit inputs and review criteria so the system can judge not only whether content is correct, but whether the result is professional and appropriate for its real use case.

Input criteria

Define a good result first

Record the audience, topic, purpose, delivery format, brand colors, and layout requirements, then use them to guide content structure and visual expression.

Check and modify

Every draft must pass a quality review

The Workflow checks facts, structure, and visual quality. If it finds imbalance, factual errors, or a standards mismatch, the draft is revised before delivery.

Relationship: TDD defines the standard the result should meet; Skills, Workflows, and Browser Operator provide the capabilities that complete the work.

05 · Three core capabilities

Three capabilities bring knowledge, successful steps, and real interactions into the system.

I / SKILLS

Load domain knowledge on demand

Skills record expert knowledge, operating steps, and quality standards. The system loads only what a task needs and turns that knowledge into steps an Agent can follow.

Expert knowledge → Executable steps
II / WORKFLOWS

Turn successful tasks into reusable workflows

The system extracts stable steps, parameters, and decision points from successful tasks, reviews them, and saves them as Workflows for similar tasks later.

Successful task → Reusable process
III / BROWSER OPERATOR

Complete page interactions in the user's browser

The system reads the current page state and performs clicks, input, and multi-step navigation in the user's signed-in context. The process remains visible and under the user's control.

Task intent → Real page interactions
06 · Agent Runtime

How the Agent Runtime executes and recovers a task.

The Agent Runtime persists task goals and state. A primary Agent decomposes work and assembles results; specialized Agents handle code, retrieval, and other tasks. Tools use a unified interface, code and files run in isolation, and durable state supports recovery after disconnects or restarts.

Task coordination

The primary Agent coordinates the task

The main Agent is responsible for decomposing tasks, scheduling sub-agents, summarizing results, and organizing task memories.

Tool calling

All tools use a unified interface

All Agents invoke tools through one interface, and results are recorded in a common event stream and task state.

Execution isolation

Process code and files in a sandbox

The sandbox is isolated from the main service; input files and output artifacts move through controlled interfaces.

My Responsibility 01 · Structured tool calling

Structured tool calling replaces parsing model-generated protocol text.

The old path required the model to emit text in a custom format for the Agent Runtime to parse. Native Tool Call uses structured fields from the model SDK for tool names, arguments, and results. The migration covered the primary Agent, sub-Agents, streaming events, result messages, and runtime telemetry, not only the parser.

Run both paths in parallel and cut over using runtime telemetry

The main Agent and sub-Agent use the same calling method, while retaining the old method as a fallback.

My Responsibility
Compatibility differences

Different model SDKs have inconsistent formats for tool call messages, streaming events, and result backfill.

Implementation method

Templates and Agent Runtime can choose structured calling (Native) or old text mode (legacy) according to configuration. The main Agent and sub-Agents use the same interface.

Migration basis

Runtime telemetry records which mode was used. Success rates and failure categories guide a gradual cutover instead of a one-time replacement.

My Responsibility 02 · Message persistence

Persist messages before notifying clients.

Ephemeral pub/sub is fast, but messages can be lost when a client disconnects or a service restarts. The new design writes message content to durable storage first; notifications only signal availability. Clients read by cursor and resume from the last recorded position after reconnecting.

message sequence

The same session is resumed in a fixed order

Message fragments for the same task use stable order; messages are isolated by session, and reading from the logged cursor continues when the client reconnects.

Migration method

Old and new transmission implementations can be switched

The unified transmission interface allows parallel, gradual switching and rollback of old and new implementations, and the business layer does not need to be rewritten with the migration.

My Responsibility 03 · User-level encryption

User-level encryption protects durable sensitive data with layered keys.

Long-running tasks require durable sensitive data, but one static key is insufficient. The design separates root keys, per-user data keys, and ciphertext format, while supporting gradual migration of existing data.

Manage keys in layers and migrate old data in stages

It is compatible with old data when reading and gradually unifies into the new ciphertext format when writing.

My Responsibility
key relationship

The root key is used to protect user-level data keys; the ciphertext metadata records key versions so that the correct version can be selected for decryption and key rotation performed.

Cipher text format

Strings and JSON (structured text format) use a unified ciphertext encapsulation format; the application program interface (API) layer calls encryption and decryption based on user context, and the business code does not directly touch the key.

Data migration

The system distinguishes plaintext from ciphertext, reads legacy data compatibly, and backfills it in batches instead of requiring a one-time migration.

My Responsibility 04 · Asynchronous task interface

Expose the Agent through a resumable asynchronous task API.

Long-running Agent work cannot depend on one synchronous request. The API creates a durable task, exposes status through subscription or polling, and returns the final result. Downstream access on a user's behalf continues to enforce that user's identity and permissions.

01

Create a persistent task

The request is written to the task store first, not just saved in the memory of the current service process.

02

Verify user authorization

The system continues to respect user identity and permission boundaries when accessing downstream resources on behalf of the user.

03

Restore state after reboot

After a restart, the service can recover in-flight tasks and continue reporting their status.

04

Return progress and results

Callers use a stable API to retrieve progress, errors, and final artifacts.
