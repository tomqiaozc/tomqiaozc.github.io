Methods · Multi-role review

Cross-check from security, architecture, and backend perspectives.

I use role-specific Agents to inspect security, architecture, and backend concerns, with an independent Agent reviewing the implementation. I define the review scope, build the minimal reproduction, verify the facts, and submit a maintainable patch.

01 / Scope of review

Determine high-risk boundaries first

Check high-risk locations such as credential storage, tool permissions, persistence, multi-tenant writes, and network egresses.

Manual focus determination
02 / Cross-check

Use separate Agents for different review perspectives

Let different agents start from different roles and assumptions, and have the results reviewed by agents not involved in the implementation.

Cover different perspectives
03 / Reproducible evidence

Explain the issue and provide tests

Organize suspicious points into problem manifestations, impacts, patches, and tests to facilitate maintainers' judgment and review.

01 · Hermes Agent

Three adopted security and reliability fixes.

Each entry explains the original problem, the change, and how maintainers adopted the implementation.

Pairing codes: replace plaintext storage with salted hashes

A pending local-gateway pairing code is a short-lived credential. If stored in plaintext, another process or tool with file access could read it directly.

Adopted after maintainer revision
Problem

The pairing code to be confirmed is saved in the local state in plaintext, and can be directly obtained by the sandbox or a process with file reading capabilities.

Change

Store a random salt plus SHA-256 hash, compare in constant time, and migrate existing plaintext records compatibly.

Public Evidence

PR #8056 ↗ · A maintainer revised the implementation and incorporated it upstream.

Detect truncated tool calls

When streaming output is truncated, finish_reason may not accurately indicate whether a tool call is structurally complete.

Adopted via cherry-picked commits

03 · Reliability discovery

Persistence failures must be detectable and recoverable.

Beyond security, I review crash safety and persistence consistency: atomic writes, transaction boundaries, error logging, and concurrency. Silent data loss can make long-running Agent tasks unrecoverable.
