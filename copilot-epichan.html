阶段一 · Copilot

控制发布风险

不可变制品、分阶段放量、功能开关、测试隔离和 Merge Queue 分别降低构建、启用、测试和合并风险。

阶段二 · Epichan

把执行要求写进系统

任务按状态推进，每个检查关卡要求证据，执行过程进入记录；失败原因可以转成新的规则或测试。

个人主责 01 · Release promotion

只把已经验证过的制品提升到下一环境。

我负责 Release promotion（制品跨环境提升）。前端制品、后端服务和运行时开关有不同的交付路径，但都遵循三项要求：使用同一份已验证制品，部署和启用分开，异常时可以快速缩小受影响范围。

BUILD

同一制品跨环境

将同一份已验证制品提升到下一环境，避免重新构建后产物发生变化。

VERIFY

进入下一环境前检查

先完成静态检查、测试和集成验证，再允许制品继续提升。

RAMP

分阶段放量

逐步扩大使用范围并观察结果，出现异常时只影响较小范围。

CONTROL

功能开关

将部署与启用分开；出现异常时，无需重新发布即可停用功能。

参与工作 · 测试稳定性

不稳定测试：识别后隔离，修复后恢复。

Flaky 测试是相同代码下可能时好时坏的测试。误报的失败过多，会让团队降低对 CI 结果的信任。处理流程需要识别可疑失败、复跑确认、保存证据，并在不阻塞其他开发工作的前提下暂时隔离用例。

给每个不稳定测试明确状态、证据和负责人

从怀疑、确认、隔离到修复和解除隔离，每一步都可以查询和追踪。

CONTRIBUTED
识别

从 CI 失败中找出疑似不稳定测试，通过受控复跑和历史结果区分真实回归与测试噪声。

隔离

暂时隔离已确认的不稳定用例并保留记录，避免单个误报持续阻塞主线。

恢复

隔离不是永久跳过。修复并验证通过后，按明确状态解除隔离。

目标：不是隐藏失败，而是区分真实回归和测试噪声，让 CI 的失败信号保持可信。

个人主责 02 · 大部分 CI 流水线迁移

CI 迁移：先验证新旧结果一致，再逐步切换。

我负责大部分流水线迁移。迁移时需要核对新旧平台的能力、缓存、权限、必需检查和触发入口，先并行验证结果，再逐项切换。Merge Queue（合并队列）则在合并前基于最新主线状态重新检查候选变更。

CI 迁移

新旧流水线并行验证

新旧流水线在一段时间内同时运行。确认关键行为和结果一致后，再逐项切换必需检查与触发入口。

个人主责 03 · 无障碍修复工作流

把无障碍修复整理成可重复的 Agent 工作流。

我负责这条无障碍修复工作流。它将问题定位、代码修改、浏览器验证和证据采集固定为重复可用的步骤，减少只靠一次临时指令处理键盘焦点、屏幕阅读器公告、ARIA 语义、高对比度和小屏布局等问题。

第二阶段 · 系统设计原则

Agent 执行系统记录任务、检查结果并保留全过程。

这里把承接需求、调度 Agent、检查结果并记录过程的一套系统称为 Harness。需求仍由人提出和确认；系统把需求转为有状态任务，并在每次状态推进前检查证据。Trace（执行记录）用于回看和审查全过程。

Task · 任务

任务有明确状态

每个任务记录当前状态、负责人、日志和外部链接，不依赖一次性聊天上下文。

Gate · 检查

通过检查才能推进

Gate（检查关卡）将权限、架构规则、评测、代码审查和 CI 设为状态推进条件。

Trace · 记录

执行过程可以追查

日志、状态变化、评测结果和截图汇总为报告，供人复核。

系统设计原则 · 任务状态

任务按明确状态推进；失败时回到对应步骤。

每个状态都要定义负责人、进入条件、所需证据和失败后的返回位置。Agent 可以执行任务，但不能自行跳过检查。

系统设计原则 · 自动检查与人工判断

模型判断语义质量，脚本检查确定规则。

代码是否符合需求通常需要模型或人判断；格式、权限、依赖方向、性能预算和测试结果则适合用脚本检查。Gate（检查关卡）组合这两类结果，避免 Agent 自行跳过交付要求。

PERMISSION

限制执行权限

只提供完成当前任务所需的权限；危险操作需要更严格的限制。

ARCH

检查依赖和结构

依赖方向、公共组件使用方式和既有架构规则由脚本检查。

QUALITY

先通过测试，再做语义审查

先运行测试和确定性检查，再判断实现是否符合需求。

PROOF

提供可复核的证据

UI 变更需要真实截图或交互记录，不能只检查代码差异。

系统设计原则 · 根据失败改进规则

每次失败都应形成可复用的规则、测试或检查。

如果只修复当前任务，同类问题仍可能再次出现。系统应记录失败原因，并将其转为可复用工作步骤（Skill）、规则、评测、架构检查或新的 Gate（检查关卡），供后续任务使用。
