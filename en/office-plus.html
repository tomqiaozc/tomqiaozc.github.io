01 · Team platform composition

Payment is only one part of a commercial delivery flow.

After the user clicks "Purchase", the backend has to deal with six types of issues: what is the product, whether the user meets the conditions, whether the payment is successful, what entitlements should be issued, how to refund or make a replacement after the delivery fails, and whether the current account and token can still be used.

Commodities and funds

Record transaction status

SKUs, pre-orders, orders, payments, refunds, and invoices together record transaction state.

Marketing activities

Evaluate campaign eligibility separately

Purchase qualifications, purchase bonuses, prize packages and prize redemption rules are separated from the order condition branch and treated as independent activity capabilities.

User entitlements

Issue or revoke benefits

Membership duration, usage-based assets and exchange entitlements define issuance, consumption and rollback rules respectively.

Long task delivery

Asynchronous execution and support for retries

Callbacks, detections, notifications, and refunds are performed through retryable background tasks and do not rely on a single request to be completed.

Accounts and sessions

Check if access is valid

Logins, tokens, device sessions, and account status collectively determine whether a user can continue to use benefits.

Result check

Find and handle status differences

Discrepancies between transactions, entitlement delivery, and external processing results can be queried and corrected through refunds, retries, or manual processing.

03 · From payment to delivery

A successful order does not mean the user has received the entitlement or result.

Orders, payments, entitlement fulfillment, and asynchronous tasks keep separate state. Idempotency keys ensure repeated requests take effect once, while durable tasks record progress. A failed step can be retried, compensated, refunded, or routed to manual handling.

Prevent repeated execution

The same business action can only take effect once

External callbacks, message re-delivery and client retries may all arrive repeatedly; idempotent keys and state transitions are used to prevent repeated deductions or repeated issuances.

Failure handling

Define recovery methods for each failure

After a third-party timeout, asynchronous task failure, or notification loss, retry, rollback, or enter manual compensation processing as appropriate.

Personal responsibility 02 · Account life cycle

Account soft deletion revokes access first while preserving recoverable state.

Deleting an account will also affect login tokens, device sessions, entitlements and risk control. Immediate physical deletion from the database will lose recovery and audit records; simply marking it as "deleted" may allow old sessions to continue to be accessed. Therefore, deletion needs to be performed step by step according to status.

Perform deletion in steps and support duplicate requests and recovery

Turn one database update into a retryable account workflow with explicit state.

My Responsibility
revoke access first

After receiving the deletion request, revoke the token, invalidate the device session, and prevent further token refreshes.

Record processing status

Clearly record the status of "requested, processing, deleted, restored"; submitting the same request repeatedly will not cause repeated side effects.

Handle concurrent requests

When deletion occurs at the same time as login or token refresh, the server account status will be used as the final judgment to prevent the account from being reactivated with old credentials.

Personal responsibility 03 · Token and device session

Token refresh, revocation, and device state must be validated together.

Access tokens provide short-lived access; refresh tokens extend a session. The change covers refresh and rotation, active revocation, device limits, expired-record cleanup, and legacy compatibility, not only token generation.

01/Token

Revoke old token after refresh

After the refresh is successful, the old credentials become invalid; the server saves the token status and can revoke it immediately when needed without waiting for natural expiration.

Limit credential validity period
02 / Equipment

Refresh token binding device records

When the device has expired, the account has been deleted, or the number of devices exceeds the limit, the server refuses to continue refreshing.

Control sessions by device
03 / Clean up

Delete expired records in batches in the background

Recoverable background tasks clean expired tokens and device records in batches to avoid processing all data in one large transaction.

Continue cleaning after failure

Migration: Support both token formats for a bounded migration period. Unify validation, refresh, and revocation first, then phase out the legacy format.

Team platform background · Channel access

Channel direct charging: Verify requests, prevent repeated fulfillment, and record callback failures.

Channel requests come from external systems. The platform first verifies the identity of the caller and whether the request has been tampered with, then accepts the request and completes the fulfillment in the background; the fulfillment results are saved separately, and can be queried and resent even if the callback of the notification channel fails.

identity

Verify channel permissions

Each channel uses clear application identity and permissions; the server confirms that the caller can perform the target operation.

sign

Verify request integrity

Combined with request signatures, validity windows, and replay protection, stop forging or resubmitting old requests.

Anti-duplication

Orders from the same channel are only fulfilled once

Use the channel order ID as the idempotency key. Repeated requests return the same business result without running fulfillment again.

Failure handling

Logging and retrying callbacks individually

The fulfillment status and callback status are persisted separately; after the callback fails, it can be retried, queried, or resent by the operator.

05 · Common engineering practices

These cases use four shared reliability practices.

record status

Save queryable status at every step

Long-running flows are split into multiple clear states; after failure, the decision is made based on the completed steps where to continue.

prevent duplication

A repeated request must not repeat the business effect

Web page request retries, message retransmissions and external callbacks may all be repeated. The business layer uses idempotent keys and status checks to avoid repeated executions.

failure compensation

Take an explicit recovery action after a cross-system failure

When payment, entitlement, and third-party state diverge, recover through retry, compensation, refund, or manual handling.

Account life cycle

Key operations check account and device status

Key operations such as logging in, refreshing tokens, or using benefits require simultaneous determination of account, token, and device status.
