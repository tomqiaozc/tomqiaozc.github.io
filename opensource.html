方法 · 多角色审查

从安全、架构和后端视角交叉检查。

我使用不同角色的 Agent 分别检查安全、架构和后端问题，并安排独立 Agent 复核实现。人负责确定审查范围、完成最小复现、确认事实，并提交可维护的补丁。

01 / 审查范围

先确定高风险边界

检查凭据存储、工具权限、持久化、多租户写入和网络出口等高风险位置。

人工确定重点
02 / 交叉审查

由不同 Agent 分别检查

让不同 Agent 从不同角色和假设出发，并由未参与实现的 Agent 复核结果。

覆盖不同视角
03 / 可复现证据

说明问题并提供测试

把可疑点整理为问题表现、影响、补丁和测试，方便维护者判断和复核。

问题 → 修复 → 测试
01 · Hermes Agent

三项被采用的安全与可靠性修复。

每项都说明原问题、修改内容和维护者采用实现的方式。

配对码：从明文存储改为加盐哈希

本地网关的待确认配对码属于短期凭据；如果明文保存，其他可读取文件的进程或工具可能直接获得它。

整理后采纳
问题

待确认配对码以明文保存在本地状态中，沙箱或具备文件读取能力的进程可直接获得。

修改

改为随机 salt + SHA-256 存储，使用常量时间比较，并兼容迁移已有的明文记录。

公开证据

PR #8056 ↗ · 维护者整理该实现后纳入主线。

检查被截断的工具调用

挑选提交后采纳

工具调用参数 JSON 被截断时，系统仍可能执行不完整的工具调用。

03 · 可靠性发现

持久化失败必须能被发现和恢复。

除安全问题外，我还检查崩溃时的数据安全和持久化一致性，包括原子写入、事务边界、错误日志和并发安全。静默丢失数据会让 Agent 长任务无法恢复。
