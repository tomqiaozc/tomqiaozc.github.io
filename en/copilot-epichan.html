Participate in work · Test stability

Flaky tests: identify, quarantine, fix, then restore.

A flaky test can pass or fail against the same code. Too many false failures erode trust in CI, so the workflow identifies suspect failures, confirms them through controlled reruns, preserves evidence, and quarantines confirmed cases without blocking unrelated work.

Give every flaky test a clear state, evidence, and owner

Every step, from suspected to confirmed, quarantined, fixed, and restored, is queryable and traceable.

CONTRIBUTED
identify

Identify suspected flaky tests from CI failures, then use controlled reruns and history to separate real regressions from noise.

isolation

Temporarily quarantine confirmed flaky tests with a recorded state so one false failure does not keep blocking the main branch.

recover

Quarantine is not a permanent skip. Restore the test after the fix passes verification.

Goal: Do not hide failures; separate real regressions from test noise and preserve trust in CI.

Phase 2 · System design principles

The Agent execution system records tasks, check results, and the full execution trace.

Here, Harness means the system that accepts a requirement, schedules Agents, checks results, and records execution. People still define and confirm requirements. The system turns them into stateful tasks and requires evidence before each transition; the Trace supports later review and audit.

Task

Tasks have a clear status

Each task records its state, owner, logs, and external links instead of relying on one chat session.

Gate · Check

Advance only after checks pass

A Gate makes permissions, architecture rules, evaluations, code review, and CI prerequisites for a state transition.

Trace·Record

The execution process can be traced

Logs, state changes, evaluation results, and screenshots are assembled into a report for human review.

System design principles · Task status

Tasks advance through explicit states; a failure returns to the relevant step.

Every state defines an owner, entry conditions, required evidence, and the return path on failure. An Agent can execute the work but cannot skip the checks.

System design principles · Automatic check and manual judgment

Models or people judge semantic quality; scripts enforce deterministic rules.

Whether code meets a requirement usually needs model or human judgment. Formatting, permissions, dependency direction, performance budgets, and test results are better checked deterministically. A Gate combines both kinds of evidence.

PERMISSION

Restrict execution permissions

Provide only the permissions needed to complete the current task; dangerous operations require tighter restrictions.

ARCH

Check dependencies and structure

Dependency direction, usage of common components, and existing architectural rules are checked by scripts.

QUALITY

Pass the test first, then do the semantic review

Run tests and deterministic checks first, and then determine whether the implementation meets the requirements.

PROOF

Provide verifiable evidence

UI changes require real screenshots or interaction records, not just checking code differences.

System design principles · Improve rules based on failure

Each failure should produce a reusable rule, test, or check.

Fixing only the current task allows the same failure to recur. The system records the cause and converts it into a reusable Skill, rule, evaluation, architecture check, or new Gate for future tasks.
